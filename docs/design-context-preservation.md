# Sub-Agent Context Preservation Design Document

## Boomerang-v3 Architecture Task — Context Preservation in Swarm

**Date**: 2026-05-19
**Status**: Design Complete — Awaiting Implementation
**Architect**: boomerang-architect (deepseek-v4-pro:cloud)

---

## Executive Summary

This document specifies a **context preservation system** that ensures sub-agent detailed findings (files read, failed attempts, rejected solutions, reasoning traces) survive in memini-ai when agents "boomerang" back to the orchestrator. The solution is a **plugin enhancement** leveraging memini-ai's existing MCP server with structured trace saving, relationship linking, and orchestrator verification.

**Key Design Decision**: Agents MUST return memory IDs of saved trace segments alongside their summary. The orchestrator verifies these IDs and passes them to subsequent agents. This creates an audit trail where "juicy details" are queryable forever.

---

## 1. Recommended Approach: Hybrid B + C + D

| Option | Verdict | Rationale |
|--------|---------|-----------|
| **A. Prompt Engineering** | Insufficient | Already in place ("save a summary"). Vague instructions produce thin saves. |
| **B. Protocol Enforcement** | **Included** | Add "DETAILED TRACE SAVE" as mandatory Step 8.1 in the 8-step protocol. Agents cannot declare completion without it. |
| **C. Structured Output** | **Primary Mechanism** | Agent MUST return `{summary, memoryIds: {...}}`. This is the **forcing function**: you cannot return IDs without saving first. |
| **D. Post-Task Hook** | **Included** | Orchestrator queries memini-ai for returned IDs to verify agent actually saved. Retries if missing. |
| **E. Automatic Indexing** | Phase 2 | Requires OpenCode-level instrumentation to intercept all agent reads/tool calls. Best long-term, not feasible now. |

**The Combination**: B ensures it's in the protocol checklist. C forces the actual save (you need IDs). D verifies it actually happened. Together they provide defense-in-depth.

## 2. Memory Schema: Relationship-Linked Trace Segments

### 2.1 Design Philosophy

Instead of one giant JSON blob or vague summaries, we use **memini-ai's native relationship graph** to link trace segments. This enables powerful queries:
- "Show me all rejected approaches for task X" (query CONTRADICTS relationships)
- "What files did boomerang-coder read before deciding on approach #3?" (follow DERIVED_FROM)
- "Find all tasks where solution A was tried and rejected" (search + relationship filter)

### 2.2 Trace Entry Types

Each agent task produces a **trace cluster** of 1-5 linked memory entries:

| Entry Type | Relationship to Master | Content |
|-----------|----------------------|---------|
| **Master Trace** | Root | Task context, final summary, agent name, timestamps, IDs of all segments |
| **Files Read** | `DERIVED_FROM` master | List of files examined with relevance notes, line ranges |
| **Approaches Tried** | `DERIVED_FROM` master | Each attempt with success/failure rationale |
| **Key Decisions** | `DERIVED_FROM` master | Decision log with reasoning, alternatives considered |
| **Rejected Code** | `CONTRADICTS` final approach + `DERIVED_FROM` master | Code that was evaluated and rejected, with why |
| **Final Implementation** | `DERIVED_FROM` master | Final code/diff, test results, validation notes |

### 2.3 Metadata Schema (per entry)

```typescript
interface TraceMetadata {
  project: "boomerang-v3";          // Always boomerang-v3
  type: "agent-trace";              // Distinguishes from regular memories
  traceType: "master" | "files-read" | "approaches" | "decisions" | "rejected-code" | "final-implementation";
  agentName: string;                // e.g., "boomerang-coder"
  taskId: string;                   // UUID generated by orchestrator
  sessionId: string;                // Current session ID
  parentMemoryId?: string;          // Master trace ID (for detail segments)
  previousTraceIds?: Record<string, string>; // IDs from prior agents in chain
}
```

### 2.4 Why Relationship-Linking?

1. **Native memini-ai support**: `create_relationship` tool exists with `DERIVED_FROM`, `CONTRADICTS`, `RELATED_TO`, `SUPERSEDES`
2. **Independent trust scoring**: Each segment gets its own trust score
3. **Granular retrieval**: Query just "decisions" without loading 10KB of file reads
4. **Contradiction detection**: Rejected vs final approaches can be explicitly linked as contradictions
5. **Evolution tracking**: `SUPERSEDES` relationships show how thinking evolved

## 3. Agent Prompt Modifications

### 3.1 Updated Mandatory Memory Protocol (All Agents)

Replace current vague Step 3:
```
3. Save when complete - Call memini-ai-dev_add_memory with a summary of your work
```

With explicit structured trace protocol:

```markdown
## MANDATORY DETAILED TRACE PROTOCOL (Step 8.1)

Before returning control, you MUST save a detailed trace to memini-ai. This preserves your reasoning for future agents.

### Step A: Save Master Trace
Call `memini-ai-dev_add_memory` with:
- **text**: Concise task summary (100 words) + list of all segment IDs you will create
- **sourceType**: "boomerang"
- **metadata**: `{project: "boomerang-v3", type: "agent-trace", traceType: "master", agentName: "<your-name>", taskId: "<from-context-package>"}`
- Save the returned `memory_id` as `masterTraceId`

### Step B: Save Detail Segments (at minimum)
For each of the following you performed during this task, save a separate memory entry:

1. **Files Read**: Call `memini-ai-dev_add_memory` with files examined, relevance notes
2. **Approaches Tried**: Call `memini-ai-dev_add_memory` with each attempt and why it succeeded/failed
3. **Key Decisions**: Call `memini-ai-dev_add_memory` with decision rationale
4. **Final Implementation**: Call `memini-ai-dev_add_memory` with final code/diff

### Step C: Link with Relationships
For each detail segment, call `memini-ai-dev_create_relationship`:
- sourceId = detail segment ID
- targetId = masterTraceId
- relationshipType = "DERIVED_FROM"

For rejected approaches vs final approach:
- relationshipType = "CONTRADICTS"

### Step D: Return Memory IDs
Your final response MUST include this JSON block:

```json
{
  "summary": "<200-word summary of what you did>",
  "memoryIds": {
    "masterTrace": "<uuid>",
    "filesRead": "<uuid or null>",
    "approaches": "<uuid or null>",
    "decisions": "<uuid or null>",
    "finalImplementation": "<uuid or null>"
  },
  "taskCompleted": true
}
```

If you cannot complete a segment (e.g., no files were read), use `null` for that ID. But you MUST save at minimum the master trace and final implementation.
```

### 3.2 Context Package Update

The orchestrator's ContextPackage will include a `traceMetadata` field:

```typescript
interface TraceMetadata {
  taskId: string;                           // UUID for this task
  sessionId: string;                        // Session ID
  previousTraceIds?: Record<string, string>; // From prior agents in chain
  traceRequired: boolean;                   // Always true for sub-agents
}
```

When dispatching agent #2, the orchestrator passes `previousTraceIds` from agent #1, so agent #2 can query detailed context directly from memini-ai instead of relying on the thin summary.

## 4. Orchestrator Modifications

### 4.1 Verification Flow (Post-Task Hook)

After an agent returns, the orchestrator performs verification:

```
Agent returns → Parse response for memoryIds JSON block
                → Query memini-ai for each ID via get_trust_score or query_memories
                → If any required ID missing:
                    → Log warning
                    → Adjust trust on agent (agent_ignored signal)
                    → Optionally retry with stronger prompt
                → If all IDs present:
                    → Adjust trust (agent_used signal)
                    → Store IDs for chain passing
                    → Add to ContextPackage for next agent
```

### 4.2 Context Package Builder Changes

In `orchestrator.ts`, the `buildContextPackage` method will:

1. Generate a `taskId` (UUID) for every dispatched task
2. Include `traceMetadata` in the ContextPackage
3. If this is agent N in a chain, include `previousTraceIds` from agent N-1

```typescript
interface ContextPackageV3 {
  // ... existing fields ...
  traceMetadata: {
    taskId: string;
    sessionId: string;
    previousTraceIds?: Record<string, string>;
    traceRequired: boolean;
  };
}
```

### 4.3 New Orchestrator Method: `verifyAgentTrace`

```typescript
private async verifyAgentTrace(agentName: string, memoryIds: Record<string, string>): Promise<TraceVerificationResult> {
  const required = ['masterTrace', 'finalImplementation'];
  const results: Record<string, boolean> = {};
  
  for (const key of required) {
    const id = memoryIds[key];
    if (!id) {
      results[key] = false;
      continue;
    }
    const score = await this.memorySystem.getTrustScore(id);
    results[key] = score !== null; // Exists in memini-ai
  }
  
  const allPresent = Object.values(results).every(v => v);
  
  if (allPresent) {
    await this.memorySystem.adjustTrust(memoryIds.masterTrace, 'agent_used');
  } else {
    await this.memorySystem.adjustTrust(memoryIds.masterTrace, 'agent_ignored');
  }
  
  return { verified: allPresent, missing: results };
}
```

## 7. Architecture Answer: Plugin + Existing MCP Server

**Question: "Would this be a plugin, an MCP server, both?"**

**Answer: Primarily a Plugin enhancement leveraging the existing memini-ai MCP server.**

- **No new MCP server needed**: memini-ai already provides `add_memory`, `create_relationship`, `query_kg`, `get_trust_score`, `adjust_trust` — all the tools required.
- **Plugin layer does the work**:
  - `orchestrator.ts` enforces trace saving via structured output parsing
  - Agent prompts instruct agents to use memini-ai tools in a structured way
  - `types.ts` defines the trace schema
- **Future Phase 2** could add an MCP middleware/server for automatic interception, but that's not required for the MVP.

The architecture is **Plugin mandates saving + Existing MCP server (memini-ai) provides structured trace schema**. This is the minimal viable change that solves the problem today.

## 8. Summary of Key Decisions

| Decision | Choice | Why |
|----------|--------|-----|
| Enforcement mechanism | Structured output + protocol + verification | Forces actual saves, verifies them, fits existing protocol |
| Trace schema | Relationship-linked segments | Uses native memini-ai graph, enables rich queries |
| Orchestrator access | Memory ID passing in ContextPackage | Direct access without re-querying by text |
| Architecture | Plugin enhancement only | No new infrastructure needed |
| First phase scope | 3 primary agents + orchestrator | Coder, architect, tester do the most complex work |
